# Brief: Cloud Connect — Insert Cloud File Links into Chat

**Status:** Draft
**Author:** Product / Architecture
**Date:** 2026-06-10
**Scope:** New first-class application "Cloud Connect" + a dedicated Settings sub-page, supporting Mail.ru Cloud, Yandex Disk, and any WebDAV-compatible storage. Files are referenced in chat by a proxy URL; no bytes are stored on 1230UI.

---

## 1. Problem statement

Today, a user who wants the agent to read a file from their personal cloud has to leave 1230UI: download the file to their device, then upload it back through the chat composer. For a Russian-speaking audience this is especially painful because the dominant personal clouds (Mail.ru, Yandex) have no general-purpose desktop-app behaviour the user can rely on.

The user-facing goal is:

> *From inside 1230UI, the user opens a file picker that lists the contents of their personal cloud, picks one or more files, and inserts a reference to each of them into the chat composer as a clickable link — without downloading them to their own machine and without 1230UI storing a copy.*

The technical goal is to do this **legally, with explicit user consent**, using a credential model that we can store, rotate and revoke, and that does not depend on undocumented APIs.

## 2. Background research

### 2.1. Available access methods per provider

| Provider | Official API | WebDAV | Auth model | Legality / TOS |
|---|---|---|---|---|
| **Mail.ru Cloud** | No public API | ✅ `https://webdav.cloud.mail.ru` | HTTP Basic with **application-specific password** generated by the user at `account.mail.ru/user/2-step-auth/passwords` | Documented and supported in `help.mail.ru/cloud_web/desktop/webdav` |
| **Yandex Disk** | ✅ REST at `cloud-api.yandex.net` | ❌ removed from current `360.yandex.ru` documentation | OAuth 2.0, refresh-token + `cloud_api:disk.read` scope | Documented at `yandex.com/dev/disk-api` |
| **Generic WebDAV** (Nextcloud, ownCloud, CloudMe, pCloud, Koofr, Synology NAS, …) | varies | ✅ | HTTP Basic / Digest / Bearer | depends on the operator |

### 2.2. Why WebDAV is the right primitive

WebDAV is a single, well-defined protocol (RFC 4918). The operations the picker needs are:

- list directory contents (`PROPFIND`, `Depth: 1`),
- stat a file / directory (`PROPFIND`, `Depth: 0`),
- issue a single signed proxy URL for `GET` (transit, never written to disk).

A single internal abstraction — `WebDavClient` — covers Mail.ru, every self-hosted cloud a user might run, and roughly a dozen public WebDAV services (rclone documents more than 30 in `rclone.org/webdav`). The Yandex adapter implements the same interface but talks REST + OAuth.

### 2.3. Why not the "guest" / public-link approach

Yandex has unofficial endpoints that resolve a `disk.yandex.ru/d/...` share to a direct download URL, and Mail.ru has a similar community-documented path (`cloud.mail.ru/public/...`). We **reject** both:

1. They depend on undocumented endpoints that break without notice.
2. They require the user to manually create a public link in the cloud UI, copy it, and paste it into 1230UI — defeating the point of the picker.
3. For Mail.ru the public-link form does not even cover private files, which is the common case.

### 2.4. Why not OAuth for Mail.ru

Mail.ru Cloud has no OAuth. The documented, supported path is WebDAV + an application password. We have to use it.

### 2.5. What we are *not* doing

- We are not downloading files to 1230UI.
- We are not validating file types, sizes, MIME, or applying any upload whitelist.
- We are not writing any bytes to disk. The proxy endpoint streams from the cloud to the agent and forgets.
- We are not implementing preview, thumbnails, or file metadata beyond what is needed to label a link.

## 3. Goals and non-goals

### 3.1. Goals

1. Ship a new application **`cloud_connect`** that lives in the existing Applications pane on the Sessions page (Task #36 infrastructure).
2. Ship a new Settings sub-page **`/settings/cloud-connect`** where the user:
   - chooses a provider preset (Mail.ru, Yandex, Nextcloud, Custom WebDAV, …),
   - enters credentials,
   - tests the connection,
   - sees existing connections and can revoke / re-test them.
3. Credentials are **encrypted at rest** in the UI DB.
4. Inside a session, the user opens the picker, navigates folders, selects one or many files, and clicks **"Insert into chat"**. The composer is populated with a Markdown link of the form:

   ```
   [📎 report.pdf · Mail.ru · 1.2 MB](https://<1230ui-host>/api/cloud/<conn>/<token>/<path>)
   ```

   Sending the message ships the link to the agent. The agent follows the link and the 1230UI backend streams the bytes **transparently** from the cloud, with the right `Content-Type` and `Content-Disposition: attachment; filename=…` headers. Nothing is persisted.
5. The agent never sees the user's cloud credentials. The link is a one-shot, short-lived signed URL scoped to a single path.
6. Cover at minimum **Mail.ru (WebDAV)** and **Yandex (OAuth)** in v1. Any other WebDAV server must be connectable via the "Custom" preset without code changes.

### 3.2. Non-goals (v1)

- Storing or caching any file bytes on 1230UI.
- File preview, thumbnails, MIME validation, size limits, upload-whitelist integration.
- Two-way sync (uploading agent output back to the cloud).
- Sharing / link generation beyond the signed proxy URL.
- Mounting multiple cloud accounts per provider simultaneously. v1 supports **one connection per (user, provider)**; multi-account is a future iteration.
- Mobile (< 1024 px). The application is `desktop_only = 1` like the rest of the applications pane.
- **Any new control inside the chat composer.** Cloud Connect lives exclusively in the Applications pane. The chat input, its file-upload button, and the surrounding chat area are not modified in any way.

The contract between the picker UI and the backend is **always** the same `CloudProvider` interface, regardless of whether the underlying transport is WebDAV or REST. This means:

- the React component never branches on provider type,
- adding a new provider is one new class + one preset, no UI or DB changes,
- the same code path serves Mail.ru, Yandex, and any user-hosted Nextcloud.

## 5. Information architecture

### 5.1. Settings → Cloud Connect page

A new top-level card on `/settings` mirroring the existing Applications / Assistants / Model Providers cards. It opens a dedicated sub-page `/settings/cloud-connect` (consistent with `/settings/providers`).

The page is organised as three vertical sections:

1. **Connections** — table of existing connections with status, "Test", "Revoke", "Remove" actions.
2. **Add connection** — a wizard (or expandable form) with:
   - Provider preset dropdown (Mail.ru Cloud / Yandex Disk / Nextcloud / Custom WebDAV).
   - Credential fields, contextual to the preset.
   - "Test connection" button.
   - "Save" button.
3. **Help / FAQ** — collapsed by default; explains where to get the Mail.ru application password, the Yandex OAuth flow, and the WebDAV URL for common self-hosted servers.

### 5.2. Sessions page → Cloud Connect application (single entry point)

In the Applications pane, a new tab **☁ Cloud Connect** is added next to **File Preview** and **File Manager** (sort order: 2, after the existing two). This is the **only** place Cloud Connect is surfaced in the application; the chat composer is left untouched.

Behaviour:

- If the user has no connection: shows a card with a single CTA "Add a cloud connection" that deep-links to `/settings/cloud-connect?returnTo=<current session>`.
- If the user has connections: shows a connection switcher (small chips at the top), then a folder browser.
- Folder browser: standard tree (path breadcrumb + lazy-loaded directory listing).
- File row: name, size, modified time, type icon, checkbox.
- **Insert into chat** — bottom-of-pane button enabled when ≥ 1 file is selected. On click, the composer is populated with one Markdown link per selected file, then the picker stays open so the user can insert more. The user explicitly hits the existing composer **Send** when ready.

Inserted links are **plain Markdown text in the composer**. They are not tracked in any database, not displayed in any file list, and carry no state of their own. If the user deletes them from the composer, they are gone.

## 6. Data model

### 6.1. New table: `cloud_connections`

```sql
CREATE TABLE IF NOT EXISTS cloud_connections (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  provider        TEXT    NOT NULL,                  -- 'mailru' | 'yandex' | 'webdav' | …
  label           TEXT    NOT NULL,                  -- user-given nickname, e.g. 'My Mail.ru'
  username        TEXT,                              -- email / login (null for OAuth-only)
  credentials_ct  TEXT    NOT NULL,                  -- AES-256-GCM ciphertext, base64
  credentials_iv  TEXT    NOT NULL,                  -- 12-byte IV, base64
  credentials_tag TEXT    NOT NULL,                  -- 16-byte GCM auth tag, base64
  config          TEXT    NOT NULL DEFAULT '{}',     -- JSON: { url, vendor, rootPath, oauthTokens, … }
  status          TEXT    NOT NULL DEFAULT 'unknown',-- 'unknown' | 'ok' | 'auth_failed' | 'network_error'
  last_tested_at  INTEGER,
  last_error      TEXT,
  created_at      INTEGER NOT NULL DEFAULT (strftime('%s','now') * 1000),
  updated_at      INTEGER NOT NULL DEFAULT (strftime('%s','now') * 1000)
);

CREATE INDEX IF NOT EXISTS idx_cloud_connections_provider
  ON cloud_connections(provider);
```

**Design decisions**

- `provider` is an internal enum-like string. The UI maps it to a display label and a credential schema; new providers do not require a migration.
- `username` is stored in cleartext because the picker UI echoes it back ("Connected as *user@example.com*") and it is not a secret on its own.
- `credentials_ct / iv / tag` together hold **one** of: `{password, refreshToken, bearerToken}` — the field used depends on `provider`. Everything else (URL, vendor, root path) lives in the unencrypted `config` JSON.
- `status` is updated by the **Test connection** action; the picker itself does not write to it. The last error is preserved for diagnostics.

### 6.2. No changes to `session_files`

Cloud Connect does not write to `session_files`, does not add columns, and does not add a `'cloud'` source. Links are just text in the composer.

### 6.3. Seed entry for the application

Extend `db/seed.js` → `seedStarterApplications()` with a third row:

```js
const cloudConnectExists = uiDb.prepare("SELECT id FROM applications WHERE key = 'cloud_connect'").get();
```

## 7. Provider abstraction

### 7.1. Interface

```ts
// lib/cloud/provider.ts
export interface CloudEntry {
  path: string;          // canonical path, e.g. '/Documents/report.pdf'
  name: string;          // basename
  isDirectory: boolean;
  size: number | null;
  modifiedAt: number | null;  // epoch ms
  mimeType: string | null;   // best-effort
}

export interface CloudProvider {
  test(): Promise<{ ok: true } | { ok: false; error: string }>;
  list(path: string): Promise<CloudEntry[]>;
  stat(path: string): Promise<CloudEntry>;
  getReadStream(path: string): { stream: NodeJS.ReadableStream; size: number | null; mimeType: string | null; filename: string };
}
```

The `getReadStream` method returns a **fresh** stream each call. The backend does not cache, does not buffer to disk, and does not memoize. If the cloud requires range requests or auth refresh, the provider handles it internally and returns a plain Node stream the route can `pipe()` directly to the response.

### 7.2. WebDAV provider

Backed by the [`webdav`](https://www.npmjs.com/package/webdav) npm package (1.x, actively maintained, ~600k weekly downloads). Constructor takes a normalised `WebDavConnection` and the in-memory credentials. The adapter translates:

- `getDirectoryContents` → `CloudEntry[]`
- `stat` → `CloudEntry` (single)
- `createReadStream(path)` → `{ stream, size, mimeType, filename }`

For Mail.ru specifically, the `vendor` hint `'nextcloud'` is set so the package enables `X-OC-Mtime` support (modified-time header), which Mail.ru also honours. This gives correct file timestamps in the picker.

### 7.3. Yandex Disk provider

Backed by `fetch()` against `cloud-api.yandex.net/v1/disk/`. Implements the same interface but with REST semantics:

- `list` → `GET /disk/resources?path=…` (paged, with `_embedded.items`).
- `stat` → `GET /resources?path=…`.
- `getReadStream` → `GET /resources/download?path=…` returns `{ href, method }`; the provider then opens a `GET` on `href` and returns the resulting `Readable` as `stream`.

OAuth tokens are stored in `config` (encrypted) and refreshed transparently on 401.

## 8. Backend API

### 8.1. `routes/cloudConnections.js`

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/cloud-connections` | list all connections (decrypted username + status only, **never** credentials) |
| POST | `/api/cloud-connections` | create a connection. Body: `{ provider, label, username, credentials, config }`. For OAuth providers, `credentials` is omitted and the server initiates the OAuth flow. |
| GET | `/api/cloud-connections/:id` | single connection (same shape as list) |
| PATCH | `/api/cloud-connections/:id` | update label / config (never credentials via PATCH; changing credentials is a separate user-initiated flow: delete + recreate) |
| POST | `/api/cloud-connections/:id/test` | invokes `provider.test()`, updates `status`, `last_tested_at`, `last_error` |
| DELETE | `/api/cloud-connections/:id` | hard delete, removes the row and forgets the encryption key ref |
| GET | `/api/cloud-connections/:id/oauth/start` | for OAuth providers, returns the authorize URL with `state` and `code_challenge` (PKCE) |
| GET | `/api/cloud-connections/oauth/callback` | handles the redirect, exchanges `code` for tokens, stores encrypted |

All routes are mounted under `apiLimiter` and require no extra auth (the UI is single-user, as the rest of the API).

### 8.2. `routes/cloudFiles.js`

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/cloud-connections/:id/list?path=/foo` | returns `CloudEntry[]` for a directory |
| GET | `/api/cloud-connections/:id/stat?path=/foo/bar.pdf` | single `CloudEntry` |
| POST | `/api/cloud-connections/:id/issue-link` | body: `{ paths: string[], ttlSeconds?: number }`. For each path, returns a signed proxy URL: `{ path, url, filename, size }`. Default TTL is 3600 s. |
| GET | `/api/cloud/<connectionId>/<token>/<encodedPath>` | **signed proxy stream**. Verifies HMAC + TTL, opens a `getReadStream` from the provider, sets `Content-Type` (best-effort from extension), `Content-Length` (if known), `Content-Disposition: attachment; filename="…"`, and pipes to the response. **Never writes to disk.** If the cloud errors mid-stream, the upstream connection is closed. |

**Link format** (issued by `/issue-link`, consumed by the proxy):

```
/api/cloud/<connectionId>/<token>/<encodedPath>
```

- `token` is `base64url(HMAC-SHA256(CLOUD_CONNECT_KEY, "<connectionId>|<path>|<expiresAt>"))` truncated to 32 chars.
- `expiresAt` is epoch seconds. The proxy rejects with `403` on mismatch or expiry.
- `encodedPath` is the canonical path with leading slash, URL-encoded.

The agent receives a regular HTTPS URL with `Content-Disposition: attachment`; no special headers or protocols required.

### 8.3. Security

- **Encryption key** is read from `process.env.CLOUD_CONNECT_KEY` (32 random bytes, base64). The helper is `lib/cloud/crypto.js`:
  - `encrypt(plaintext)` → `{ ct, iv, tag }` (AES-256-GCM, 12-byte IV per call).
  - `decrypt({ ct, iv, tag })` → `string`.
  - On first startup, if the key is missing, the server logs a one-time warning and refuses to create connections (existing rows still work if the key is in place, otherwise they are listed as `status = 'unknown'` with `last_error = 'ENCRYPTION_KEY_MISSING'`).
- **HMAC key** for signed proxy URLs is derived from the same `CLOUD_CONNECT_KEY` (`HKDF` with info string `'cloud-proxy'`). This way operators configure exactly one secret.
- **Transport**: 1230UI is intended to run behind HTTPS in production; the UI shows a warning if `location.protocol === 'http:'` and a cloud connection is being added.
- **OAuth callbacks** are server-side and terminate on our own origin. No third-party redirect trust issues.
- **Signed-link scope**: a token is bound to a specific `(connectionId, path, expiresAt)` triple. Tampering with the path invalidates the HMAC. Brute force is infeasible (256-bit key, 32-char truncated token).
- **Audit log**: every connection create / test / delete writes a row to a new `cloud_audit` table (id, connection_id, action, ip, ua, created_at). Signed-link usage is logged with `connection_id`, `path`, `bytes_sent`, `status`. Not exposed in UI in v1; available for diagnostics.

`issueLinksForSelected` calls `POST /api/cloud-connections/:id/issue-link` once with all selected paths, and returns `{ path, url, filename, size }[]`.

### 9.4. Inserting into the chat

The composer is not modified. The picker instead **calls back into the existing chat store** with a string to append at the current cursor position. Specifically, `cloudConnectStore.issueLinksForSelected()` returns Markdown links, and the application component dispatches an `appendToComposer(text)` action on the existing `useChatStore` (Zustand) — same store the local file upload already uses to insert file references. If that action does not exist, the brief adds it as a single new action; nothing else in the composer changes.

Each inserted link looks like:

```markdown
[📎 report.pdf · Mail.ru · 1.2 MB](https://<host>/api/cloud/3/aB12…/Documents%2Freport.pdf)
```

Separator between multiple inserts: two spaces + newline (standard Markdown line break). The user can edit freely after insertion.

## 11. Error handling matrix

| Condition | UI behaviour |
|---|---|
| 401 from cloud (revoked password) | Connection row turns red, status `auth_failed`. Picker shows a banner: "Your Mail.ru password is no longer valid. [Reconnect]." Reconnect opens `/settings/cloud-connect` with the connection preselected. |
| Network timeout | Banner: "Couldn't reach Mail.ru. [Retry]". The cached listing (if any) stays visible. |
| Signed link expired / tampered | Proxy returns `403`. The agent (or the user, if they follow the link) gets a clean error. Picker is unaffected. |
| OAuth state mismatch | Callback page shows "Authentication failed. Please try again." and the user returns to the empty form. |
| `CLOUD_CONNECT_KEY` missing | All "Add connection" UI is disabled; banner: "Server is missing its encryption key. Set `CLOUD_CONNECT_KEY` in the environment." |

We deliberately do **not** validate file type, size, or MIME on insert. If the cloud returns an error mid-stream, the proxy closes the upstream connection and returns `502` to the agent — same semantics as a slow / broken download. The picker itself never has to deal with file content.

- **`webdav`** ^5.x — WebDAV client for Node, ESM, actively maintained. ~600k weekly downloads.

## 13. Configuration

Add to `config.js`:

```js
cloudConnect: {
  encryptionKeyEnv: 'CLOUD_CONNECT_KEY',   // 32 random bytes, base64 (also used for HMAC via HKDF)
  linkTtlSeconds:   3600,                  // default signed-URL TTL
  requestTimeoutMs: 30_000,
  listingPageSize:  200,
  oauthCallbackUrl: '/api/cloud-connections/oauth/callback',
},
```

Operator must generate the key once:

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
# → paste the output as CLOUD_CONNECT_KEY in .env
```

The CHANGELOG and `docs/INSTALLATION.md` will be updated to call this out.

## 14. Migration / rollout

1. Deploy the schema change. `initSchema()` is idempotent and `cloud_connections` is additive.
2. Deploy the backend. Existing functionality is unaffected.
3. Generate `CLOUD_CONNECT_KEY` in the environment.
4. Restart. The application appears in the Applications pane and the new card appears on `/settings`.
5. Users can opt in by adding a connection at their own pace. There is no forced migration; v1 ships with no automatic connection of any account.

## 15. Out of scope (future work)

## 16. Resolved questions

1. **Encryption key rotation policy.** *Out of scope for v1.* Documented below as a known operational caveat; UI for rotation is a future task.

   `CLOUD_CONNECT_KEY` serves two purposes: (a) AES-256-GCM key for encrypting stored cloud credentials in the UI DB, and (b) HKDF-derived HMAC key for signing proxy URLs. Both are bound to a single secret in `.env`. There is no rotation flow in v1; if the key is replaced, all stored connections become undecryptable and all currently-issued signed links become unverifiable (the agent's pending downloads fail with `403`). Operators who must rotate today do so by generating a new key, restarting the service, and accepting that every user will be prompted to re-enter their cloud credentials on next use. This is acceptable for v1 because the key is operator-controlled, not user-controlled, and is never exposed in UI or API responses.

2. **Yandex OAuth app.** *Open.* The current docs require registering an app at `oauth.yandex.com/client/new` with a `redirect_uri` pointing to our callback. The integration is implementable in v1 but blocks on the availability of a stable public URL for the deployment.

3. **Self-hosted defaults.** *Mail.ru + Yandex + Custom WebDAV only.* Nextcloud / ownCloud are reachable via the Custom WebDAV preset; dedicated presets are deferred.

4. **Link TTL.** *1 hour.* The proxy endpoint rejects expired links with `403`. Per-request override is exposed via `?ttlSeconds=N` on `/issue-link` for future use cases, but the UI in v1 always requests the default. The chosen value gives agents enough headroom for queueing and retries within a typical chat turn, while still bounding the blast radius if a link leaks.
