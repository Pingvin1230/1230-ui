## 1. UX/UI аудит

### Общая оценка

Платформа находится на хорошем уровне для pre-release продукта. Видна чёткая дизайн-система (CSS-переменные, дизайн-токены), мобильная адаптация сделана осознанно. Основные проблемы — в согласованности между страницами и в том, что несколько критически важных для нетехнических пользователей паттернов ещё отсутствуют.

---

### 1.1 Сильные стороны

- **Дизайн-система.** Единая система токенов (`--color-bg-primary`, `--color-accent` и т.д.) в `index.css`. Тёмная и светлая темы переключаются без артефактов. JetBrains Mono для кода — правильный выбор.
- **Мобильная адаптация.** Touch-targets 44×44 px, `env(safe-area-inset-bottom)` для iOS, swipe-to-delete, long-press для bulk mode, `clamp(14px, 0.5vw + 13px, 16px)` для fluid typography — всё сделано грамотно.
- **Пустые состояния.** SVG-иллюстрации для пустых списков есть, что снижает дезориентацию пользователя.
- **Обратная связь в процессе.** Индикаторы «Agent is thinking» / «Generating response» / «Executing tool» дают пользователю понять, что происходит. Toast-уведомления с очередью и auto-dismiss.
- **Виртуализация.** `react-virtuoso` для списка сессий — правильное решение для масштабирования.

---

### 1.2 Проблемы и рекомендации

#### P1 — Критично (блокируют цель «нетехнический пользователь»)

**[UX-1] Нет загрузки файлов.**  
Задача #23 в TODO. Для большинства реальных сценариев (анализ логов, ревью кода, Q&A по документу) файловый ввод обязателен. Пользователь вынужден копировать содержимое вручную — это friction-killer.  
*Рекомендация:* Приоритизировать в v1.0.0. Минимум: drag-and-drop + file picker + чипы прикреплённых файлов над полем ввода.

**[UX-2] Нет предпросмотра файлов из ответа агента.**  
Задача #24. Агент говорит «I saved it to /tmp/report.md», пользователь не может этого увидеть не выходя из UI.  
*Рекомендация:* File-карточки в сообщении агента с inline-просмотром и кнопкой Download.

**[UX-3] Термины «model id», «API key», «provider» видны в UI.**  
В Settings и на странице Providers нетехнический пользователь встречает сырую терминологию инфраструктуры.  
*Рекомендация:* Переименовать «Model Providers» → «Подключённые сервисы», «API key» → «Ключ доступа», добавить однострочные описания для каждого провайдера на ProvidersPage.

---

#### P2 — Важно

**[UX-4] Нет подтверждения сохранения в ChatPage при уходе со страницы.**  
Если пользователь начал набирать сообщение и случайно кликнул на другую сессию — текст теряется без предупреждения.  
*Рекомендация:* `beforeunload` / `useBlocker` (React Router) с диалогом «Уйти? Введённый текст будет потерян».

**[UX-5] Dashboard — Quick Chat создаёт сессию с заголовком из первых 60 символов сообщения.**  
Это работает, но заголовок получается неинформативным для длинных запросов. Пользователь не может изменить название до создания сессии.  
*Рекомендация:* Реализовать Task #0 (Smart session titles via LLM) в приоритетном порядке, либо показывать поле «Название сессии» как опциональное на Dashboard.

**[UX-6] Нет визуального разделения между «ассистент» и «модель» для новых пользователей.**  
На `/new` тайлы ассистентов и стандартный тайл выглядят одинаково, но несут разную семантику.  
*Рекомендация:* Добавить секционный заголовок «Ассистенты» над плитками ассистентов и «Быстрый старт» над стандартным тайлом.

**[UX-7] Отсутствует onboarding / welcome flow.**  
Новый пользователь после установки видит Dashboard с пустым состоянием, но не понимает, что делать дальше.  
*Рекомендация:* Минимальный onboarding: 3-4 шага (установлен ли ключ провайдера → выбери модель → создай первую сессию). Можно реализовать как dismissable banner.

**[UX-8] Bulk actions в SessionsPage доступны только после long-press на мобильном, что неочевидно.**  
Affordance не раскрыт — нет никакой подсказки о существовании этой функции.  
*Рекомендация:* Tooltip или подсказка «Hold to select» на session card при первом использовании.

---

#### P3 — Улучшение

**[UX-9] ChatPage — кнопки Copy/Regenerate появляются при hover, но на мобильных всегда видимы.**  
Разрыв в поведении между desktop и mobile — разный UX для одного и того же пользователя при смене устройства. На десктопе кнопки скрыты до hover, что снижает discoverability.  
*Рекомендация:* Рассмотреть постоянную видимость кнопок с уменьшенной opacity (как уже сделано на мобильном).

**[UX-10] В Settings слишком много разделов на одной странице без навигации.**  
SettingsPage (~785 строк) содержит General, Models, Commands, Hermes Status, About — всё в одном скролле.  
*Рекомендация:* Разбить на подразделы с якорями или на отдельные sub-routes (уже есть прецедент — `/settings/providers`).

**[UX-11] Нет счётчика токенов в процессе ввода.**  
Пользователь не знает, насколько близко к лимиту модели его сообщение.  
*Рекомендация:* Лёгкий token count estimate под textarea (опциональный, только когда контекст > 70% лимита).

**[UX-12] Отсутствует визуальный индикатор текущей версии в шапке.**  
Есть в Settings, но сложно найти новому пользователю.

### 1.3 Навигация и информационная архитектура

```
Текущая IA:
Dashboard → Sessions → Chat
              ↓
           New Session (с ассистентами)
              ↓
```

*Рекомендация:* Добавить прямую ссылку на `/assistants` в Sidebar и MobileNav.

## 2. Аудит программного кода

### Общая оценка

Код написан качественно для своего масштаба. 100% TypeScript на фронтенде, zod-валидация конфига, разумное разделение ответственности. Главные системные риски — в архитектуре бэкенда: монолитный `server.js` на 1871 строку и отсутствие тестов.

---

### 2.1 Сильные стороны

- **TypeScript 100% на фронтенде.** Строгая типизация, отдельные `types/api.ts`.
- **Zustand для стейт-менеджмента.** Минимальный и предсказуемый. 8 сторов с чёткой ответственностью.
- **Lazy-loading всех страниц.** `React.lazy()` + `Suspense` + `PageSkeleton` — правильный паттерн.
- **Безопасность.** helmet, CORS, rate limiting (5 разных лимитеров), XSS sanitization, секреты никогда не возвращаются в API.
- **Транзакционность в SQLite.** Fork-on-edit для ассистентов завёрнут в транзакцию — корректно.
- **WAL mode + busy_timeout** для SQLite writable connection — правильно.
- **Structured JSON logging** в middleware.
- **Идемпотентные миграции** через `PRAGMA table_info` вместо `ALTER TABLE IF NOT EXISTS` — корректный обходной путь для SQLite.
- **Code splitting:** Dashboard 6 KB · Sessions 80 KB · Settings 21 KB — разумное разбиение.

---

### 2.2 Проблемы и рекомендации

#### P1 — Архитектурные риски

**[CODE-1] `server.js` — монолит на 1871 строку.**  
Всё в одном файле: инициализация БД, миграции, seeder, 20+ маршрутов, вспомогательные функции, SSE-стриминг. Файл невозможно тестировать по частям, изменение одного маршрута требует работы со всем контекстом.  
*Рекомендация:* Разделить на модули: `routes/sessions.js`, `routes/chat.js`, `routes/models.js`, `routes/assistants.js`, `routes/providers.js`, `db/init.js`, `db/migrate.js`, `db/seed.js`. Это крупный рефактор, но откладывать его дальше будет дороже.

**[CODE-2] Нет тестов (0% coverage).**  
Тест-coverage = 0%, что явно зафиксировано в TODO. При текущем темпе разработки (6 релизов за 4 дня) регрессии неизбежны.  
*Рекомендация:* Начать с backend unit tests (Tasks #11) для критических эндпоинтов: `/api/sessions`, `/api/chat`, `/api/models`. Vitest предпочтительнее Jest из-за совместимости с ESM.

**[CODE-3] Две соединения к Hermes DB (`db` readonly + `hermesDbWrite`) — неочевидный риск.**  
`db` открыт как readonly для чтения, `hermesDbWrite` — writable для удаления сессий. Оба открыты одновременно на протяжении всего времени жизни процесса. WAL-режим включён только для `hermesDbWrite`.  
*Рекомендация:* Задокументировать это решение. Рассмотреть единое соединение с WAL — SQLite корректно обрабатывает concurrent reads с WAL без readonly-хака.

---

#### P2 — Качество и надёжность

**[CODE-4] `execSync` для получения версии Hermes в `/api/system/status`.**  
`execSync('hermes --version')` в обработчике HTTP-запроса блокирует Node.js event loop на время выполнения системной команды.  
```javascript
// server.js ~264
const versionOutput = execSync('hermes --version', { encoding: 'utf-8' });
```
*Рекомендация:* Заменить на `execFileAsync` / `spawnAsync` (промис-обёртка над `child_process.spawn`) или вынести в фоновый процесс с кешированием результата.

**[CODE-5] Незакрытые соединения с БД при ошибке старта.**  
Если открытие `db` прошло успешно, но открытие `uiDb` упало — `db` не закрывается перед `process.exit(1)`.  
*Рекомендация:* Добавить cleanup handler или обернуть в try-finally.

**[CODE-6] `useEffect` с неполными зависимостями в ChatPage — известный баг.**  
Зафиксирован в Task #26: `useEffect` на строке 152 в `ChatPage.tsx` эмитит `react-hooks/exhaustive-deps` warning.  
*Рекомендация:* Быстрое исправление (S-сложность), нужно закрыть до v1.0.0.

**[CODE-7] `formatTimestamp` дублируется в SettingsPage.**  
Зафиксирован в Task #27. `SettingsPage.tsx:238` содержит локальный хелпер, дублирующий логику `lib/time.ts`.  
*Рекомендация:* Быстрое исправление (30 мин). Расширить `lib/time.ts` → удалить локальный хелпер.

**[CODE-8] `package.json` версия (0.5.1) расходится с README (0.6.1).**  
`package.json` не обновлялся при выпуске v0.5.2–v0.6.1.  
*Рекомендация:* Синхронизировать `package.json#version` с тегом релиза. Рассмотреть автоматизацию через `npm version patch/minor` в CI/CD.

**[CODE-9] `geoip-lite` в зависимостях production.**  
`geoip-lite` (~30 MB данных MaxMind) нужен только для фичи «Like» — Mattermost webhook с геолокацией. Это тяжёлая зависимость для одного декоративного endpoint.  
*Рекомендация:* Перенести в optional dependency или убрать гео-логику из лайк-фичи.

**[CODE-10] Нет CI/CD (Task #17).**  
Отсутствие автоматического lint + typecheck + build на PR означает, что сломанный код может попасть в main.  
*Рекомендация:* Минимальный GitHub Actions workflow: `npm run lint && npm run build` на каждый push в main/PR.

---

#### P3 — Мелкие улучшения

**[CODE-11] `sanitizeBody` в `middleware/security.js` не обрабатывает вложенные объекты рекурсивно.**  
```javascript
// middleware/security.js:61-74
// Только один уровень вглубь — вложенные объекты не санируются
result[key] = value; // если value = { nested: '<script>...' }
```
*Рекомендация:* Рекурсивная санация или использование готовой библиотеки.

**[CODE-12] GitHub API URL захардкожен в `server.js`.**  
```javascript
// server.js ~299
const githubResponse = await fetch('https://api.github.com/repos/NousResearch/hermes-agent/releases/latest', ...
```
Это URL Hermes Agent, а не 1230UI — сделано намеренно, но стоит вынести в config.js.

**[CODE-13] `assistantColors.ts` — mapping цветов только для 8 значений, но без fallback.**  
Если цвет не найден в map — компонент получит `undefined` и, возможно, рендерит без стилей. Нужен fallback-цвет.

**[CODE-14] Отсутствуют типы для `server.js`.**  
Бэкенд написан на чистом JS. Учитывая, что фронтенд полностью на TypeScript и зависимости для `@types/*` уже установлены (`@types/node`, `@types/better-sqlite3`), конвертация бэкенда в TypeScript была бы логичным следующим шагом.

### 2.3 Архитектура данных

**Сильные стороны:**
- Два раздельных файла БД (Hermes DB readonly + UI DB readwrite) — правильная изоляция.
- Foreign keys с `ON DELETE SET NULL` для `session_meta.assistant_id` — корректно.
- Index покрытие: `idx_assistants_archived`, `idx_assistants_model`, `idx_session_meta_assistant`, `idx_likes_user_hash` — есть.

**Слабые места:**
- Нет FTS5 индекса для поиска по сессиям (Task #13) — клиентский поиск не масштабируется.
- Нет versioned миграций (нет нумерованных migration-файлов) — только ad-hoc `PRAGMA table_info` checks. При росте схемы это станет проблемой.

## 3. Возможности развития продукта

### 3.1 Ближний горизонт (v1.0 → v1.x)

- **Файловый ввод и вывод** (Tasks #23, #24) — разблокирует большинство реальных use cases.
- **Пресеты с параметрами модели** — Phase 2 ассистентов: system prompt, temperature, max_tokens. Инфраструктура готова, нужна UI-часть.
- **Умные заголовки сессий** — LLM-генерация заголовка из первого сообщения. Низкая сложность, высокий UX-эффект.
- **Session Export** (Markdown/JSON) — даёт пользователям контроль над своими данными.
- **Per-message actions** — копирование, регенерация, «объясни это», «переведи» — расширяет полезность каждого ответа.

### 3.2 Средний горизонт (v2.x)

- **Full-text search по контенту сессий** (FTS5 + Task #13) — критично при накоплении 100+ сессий.
- **Multi-user mode** — изоляция сессий по пользователям, session ownership. Открывает командное использование.
- **Docker** (Task #18) — снижает барьер установки для DevOps-сценариев.
- **Webhooks / внешние интеграции** (Telegram, Slack) — агент как сервис вне браузера.
- **Backup/Restore** БД — обязательно для production-использования.

### 3.3 Дальний горизонт (v3+ / post-1.0)

- **Multi-Agent Federation** (Task #21) — несколько Hermes-агентов на разных серверах, управляемых из единого UI. Уже есть детальный дизайн-документ (`FEDERATION_DESIGN.md`). Открывает позиционирование как «multi-agent OS».
- **Marketplace ассистентов** — импорт/экспорт конфигураций ассистентов, sharing между пользователями и экземплярами.
- **Аналитика использования** — статистика по сессиям, моделям, времени отклика — ценно для enterprise-сценариев.
- **Plugin/tool ecosystem** — расширяемость через кастомные tool-handlers для Hermes.

### 3.4 Позиционирование продукта

Продукт находится на пересечении двух ниш:

1. **Personal AI assistant UI** — конкурирует с Open WebUI, Lobe Chat, AnythingLLM.
2. **DevOps AI terminal** — управление несколькими агентами на нескольких серверах.

Уникальное преимущество 1230UI — глубокая интеграция с Hermes Agent (tool calls визуализация, статус агента, управление провайдерами) при сохранении простоты для нетехнического пользователя. Federation-дизайн — сильный дифференциатор, если будет реализован.
